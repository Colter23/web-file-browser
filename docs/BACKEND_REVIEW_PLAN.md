# 后端分批审查清单

本文档用于固定后端后续审查顺序。审查时优先发现真实影响安全、正确性、性能或可维护性的离散问题；不属于边界 bug 的设计问题先记录为后续统一优化项，避免每轮审查扩大成大重构。

## 审查原则

- 每轮只审一个风险域，控制范围，先发现问题，再决定是否修复。
- 问题分为两类：
  - 必须修复：会影响路径安全、文件正确性、性能边界、认证保护或数据一致性。
  - 后续优化：接口设计、模块边界、命名、重复逻辑、可测试性等设计债。
- 审查结论需要落到具体文件、函数和场景。
- 后续开发仍以 Linux Docker、局域网、单管理员、性能优先、轻量配置为边界。

## 审查顺序

1. 路径与挂载安全
   - 范围：`path_resolver`、`mapping_store`、`file_ops`，以及调用这些能力的写入入口。
   - 目标：确认不能 `..` 越界、不能通过符号链接逃逸、不能跨挂载写入、只读挂载确实只读、挂载根保护有效。

2. 大文件传输性能
   - 范围：`transfer.rs`、`routes/file.rs`、`request_limits.rs`。
   - 目标：确认上传、下载、Range、保存编辑都是流式；没有整文件读入内存；并发限制和大小限制生效。

3. 目录浏览性能
   - 范围：`path_resolver/listing.rs`、`routes/file.rs`。
   - 目标：确认 1 万文件目录下分页、排序、过滤、`detail=basic/full` 不做额外重型 `stat`；默认接口不会全量返回总数。

4. 删除、回收站、直接删除
   - 范围：`trash.rs`、`routes/trash.rs`、`file_ops.rs`。
   - 目标：确认默认进回收站、可直接删除、批量操作部分失败可追踪；避免跨盘大文件无意义搬入中心回收站。

5. 后台任务与压缩解压
   - 范围：`task_io.rs`、`tasks.rs`、`routes/tasks.rs`。
   - 目标：确认复制、移动、压缩、解压不会阻塞异步运行时；支持取消；防止 Zip Slip；大文件复制保持流式；任务状态不会无限增长。

6. 配置热更新
   - 范围：`config.rs`、`settings.rs`、`routes/settings.rs`。
   - 目标：确认运行配置和启动配置边界清晰；环境变量锁定逻辑正确；更新后只影响后续请求；不在热路径读盘。

7. 认证与 API 错误
   - 范围：`auth.rs`、`auth_store.rs`、`error.rs`、`routes/mod.rs`。
   - 目标：确认未登录保护完整，错误结构稳定，前端国际化 `reason/params` 可依赖，登录态不会误放行敏感接口。

8. 搜索、收藏、审计、健康检查
   - 范围：`search.rs`、`favorites.rs`、`audit.rs`、`ops.rs`。
   - 目标：确认增强能力不引入无意义扫盘、日志无限增长或接口语义不一致。

9. 启动、部署与静态资源就绪
   - 范围：`main.rs`、`config.rs`、`app.rs`、`ops.rs`、`Dockerfile`、打包脚本和部署文档。
   - 目标：确认 Linux Docker、Windows 测试包、健康检查、监听地址解析和静态资源托管不会出现“进程可启动但实际不可用”的部署问题。

## 审查记录

### 第一轮：路径与挂载安全

已处理：

- 修复冲突检测跟随符号链接的问题：新建、上传、后台任务输出、回收站恢复等入口现在会把符号链接视为已占用路径。
- 显式覆盖文件时，如果目标是符号链接，会直接拒绝，不再把它当作普通文件处理。
- 增加测试覆盖：挂载目录中存在指向挂载外缺失文件的符号链接时，新建文件不能跟随该链接写到挂载外。

后续统一优化记录：

- 当前写入目标主要通过“先解析父目录，再写入子路径”保证边界；如果挂载目录同时被外部本机用户高频修改，仍存在常见文件系统 TOCTOU 竞态。局域网单管理员边界内暂不引入 `openat`/`O_NOFOLLOW` 这类复杂实现，后续如需要更强本机多用户隔离再评估。
- 读取类接口会跟随符号链接，但 canonical 后必须仍在挂载根内；删除、移动、复制、压缩等写入/破坏性操作会拒绝符号链接。后续可以统一评估是否改为“所有符号链接都不可访问”，但这会降低可用性。
- 后台复制挂载根目录时，默认输出名称来自真实本地目录名，而不是虚拟挂载名。这不是边界 bug，但属于接口语义和用户体验可优化项。

### 第二轮：大文件传输性能

审查结论：

- 未发现必须立即修复的传输性能 bug。
- 下载和预览使用 `tokio::fs::File`、`ReaderStream` 和 `take(length)` 流式响应，未发现整文件读入内存。
- `GET /api/content/*` 与 `GET /api/download/*` 的响应体持有传输并发许可，客户端未读完或响应体未释放时不会提前释放许可；已有 API 测试覆盖。
- 普通 `HEAD /api/content/*` 和 `HEAD /api/download/*` 只读取文件元数据，不打开文件句柄；编辑模式 HEAD 仍会执行文本安全探测，符合编辑保护语义。
- 上传和保存使用请求体或 multipart field 分块写入临时文件，超限或失败会清理临时文件；保存接口会二次校验 `If-Match` 后再替换目标文件。
- Range 解析只支持单段，非法或越界 Range 返回稳定 `416`；已有单元测试和 API 测试覆盖。

后续统一优化记录：

- HEAD 当前仍会占用一次传输并发许可，虽然生命周期很短；如果后续高频 HEAD 探测很多，可以考虑单独的轻量限流或跳过传输许可。
- 编辑保存会逐块检查 NUL 字节，属于线性 CPU 成本；由于编辑接口已有 `maxEditBytes` 上限，当前不需要复杂化。
- multipart 上传多个文件时已改为逐文件结果：前序文件成功后，后续文件失败会返回 `207 Multi-Status` 和 `files/errors/success/failed`，明确展示部分成功；单文件失败仍保持原来的 4xx 错误。
- 上传限额按单个文件 field 计算，不是整个 multipart 请求总量。局域网单管理员边界下暂不增加全请求总量计数；如以后需要防止一次请求上传大量文件，再补请求级总量限制。

### 第三轮：目录浏览性能

已处理：

- 修复目录查询校验被条件请求预检查绕过的问题：`detail=basic&sort=size/modified` 现在会在路由层先返回 `400 BAD_REQUEST`，即使请求带有命中的 `If-Modified-Since`，也不会误返回 `304`。
- 增加 API 回归测试覆盖上述场景，避免缓存预检查再次跑在参数校验之前。

审查结论：

- 默认目录接口会使用 `maxDirPageSize` 作为分页上限，未传 `limit` 时也不会全量返回目录内容。
- `detail=basic + sort=name` 使用排序窗口保留当前页所需条目，避免对 1 万级目录保存和排序全部结果；为保证排序和 `hasMore` 正确，仍需要完整遍历目录。
- `detail=full + sort=name` 只对当前页窗口做元数据读取；`sort=size/modified` 明确要求 `detail=full`，会对匹配条目读取元数据后再排序，属于显式昂贵操作。
- 默认不返回 `folderTotal/fileTotal`；当前总数来自同一次目录遍历，不会为了默认响应额外扫盘或二次读取。
- `detail=basic` 的 `If-Modified-Since` 预检查只读取目标自身元数据，命中时可在目录扫描并发许可和目录遍历前返回 `304`。

后续统一优化记录：

- 带 `If-None-Match` 的目录条件请求仍需要生成响应 JSON 后计算 ETag；这是精确 ETag 的代价。轻量项目当前接受，后续如目录很大且客户端频繁使用 ETag，可以考虑基于目录 mtime 和查询参数生成弱 ETag。
- 虚拟目录来源于挂载配置，规模通常很小，目前仍是先收集排序再分页；如未来挂载点数量大幅增加，再考虑统一使用窗口排序。
- 目录列表用轻量 `DirEntry::file_type()` 分类；符号链接目录可能在列表里显示为文件，但进入路径时会按真实目标解析。该问题偏语义一致性，不属于本轮目录性能缺陷，后续可结合符号链接访问策略统一优化。

### 第四轮：删除、回收站、直接删除

已处理：

- 修复回收站清空的并发一致性问题：`POST /api/trash/empty` 现在只移除本次清空开始时选中的记录，避免清空过程中新增的回收站记录被误从索引中删除。
- 修复回收站 payload 与记录元数据文件名碰撞问题：新回收站条目的文件内容统一放入 `payload/` 子目录，避免用户删除名为 `metadata.json` 的文件时覆盖记录元数据。
- 保持旧回收站记录兼容：清理和恢复仍支持历史的 `{id}/{原文件名}` 布局，新条目使用 `{id}/payload/{原文件名}`。
- 调整删除到回收站的落盘顺序：先写单条 `metadata.json`，再移动源文件；如果移动失败，会清理本次创建的记录目录，减少“源文件已离开原位但记录未写好”的窗口。
- 更新 API 文档中的 `trashPath` 示例，并增加单元测试覆盖 `metadata.json` 文件名碰撞和新旧布局清理。

审查结论：

- 普通删除默认进入回收站，直接删除需显式 `permanent=true`；两条路径都会复用 `resolve_delete_target`，保持只读挂载、挂载根、内部回收站目录和符号链接目标保护。
- 删除到回收站优先使用挂载根内 `.web-file-browser-trash/`，不可用时回退中央 `data/trash/`，符合 Linux Docker 下减少跨挂载复制的性能边界。
- 单条恢复、批量恢复、单条清理、批量清理都有 API 测试覆盖；批量接口保留部分成功和逐条错误信息。
- 回收站保留策略启动时只读索引，不主动扫描 payload；容量策略优先使用记录的 `sizeBytes`，不会为了普通列表或启动额外统计大目录。

后续统一优化记录：

- `empty` 和保留策略清理在某条记录清理失败时会返回错误，已删除的前序 payload 可能仍保留索引记录；当前偏保守，避免误删索引。后续如需要更好的运维体验，可以改成批量结果结构。
- 文件夹删除在同盘 `rename` 成功时会整体移动目录，包括内部符号链接；跨盘回退复制时会拒绝内部符号链接。这属于符号链接策略一致性问题，不是本轮回收站性能缺陷，后续可和全局符号链接策略一起收敛。
- 回收站索引仍是单个 JSON 文件，轻量边界内可接受；如果未来回收站记录数量明显变大，再考虑 JSONL 或 SQLite 记录历史。

### 第五轮：后台任务与压缩解压

已处理：

- 修复后台复制、压缩和解压在“临时输出已写好，但最终刷新进度或落盘替换失败”时可能遗留隐藏临时文件的问题；现在统一通过 `flush_and_finalize_temp_path` 收尾，任一步失败都会清理本次临时路径。
- 增加单元测试覆盖任务取消导致最终刷新失败、目标替换失败两种收尾失败场景，确保临时输出不会残留。

审查结论：

- 复制、压缩和解压中的阻塞文件系统工作都放在 `spawn_blocking` 中执行，不会长时间占用异步运行时工作线程。
- 大文件复制、压缩和解压使用固定大小缓冲区分块读写，未发现整文件读入内存。
- 压缩源和递归条目会拒绝符号链接和特殊文件；解压会拒绝 Zip Slip、反斜杠路径、过深路径、符号链接和不支持的 tar 条目类型。
- 任务状态为内存保留，已结束任务按 `taskHistoryLimit` 裁剪，排队和运行中的任务不会被历史清理移除。
- 取消检查覆盖排队任务、批量任务循环、复制/压缩/解压读写循环和限速等待；取消或失败时临时输出会被清理，覆盖目标会保留旧文件直到新文件完整落盘。

后续统一优化记录：

- 后台复制、解压目录完成后，搜索索引只增量更新输出根路径，不递归扫描新目录子项；这是性能边界下避免无意义扫盘的取舍。如后续需要更完整的搜索体验，可增加显式“任务完成后索引新目录树”的可配置选项，默认仍不扫描。
- 移动任务复用同步移动能力，当前没有逐字节进度；同挂载内移动通常是快速 `rename`，符合轻量边界。若以后支持跨挂载后台移动，再改为流式复制加删除并提供字节进度。
- 压缩任务输出名称在复制或压缩挂载根目录时仍来自真实本地目录名，不一定等于虚拟挂载名；这属于接口语义优化，不是当前后台任务正确性问题。

### 第六轮：配置热更新

已处理：

- 修复 `PATCH /api/settings` 同时提交 `runtime` 和 `startup` 时可能部分生效的问题：现在会先校验两边补丁，再写一次 `config.json`，最后统一替换内存快照；任一部分失败时不会热应用 runtime，也不会写入配置文件。
- 修复配置写盘会全量写入内存快照的问题：现在只写本次请求实际触碰的字段，避免把环境变量覆盖后的运行值旁路写回 `config.json`。
- 收敛 `SettingsService` 更新入口，路由层和服务层测试都使用统一的事务式 `patch` 方法，减少多入口语义漂移。
- 增加 API 回归测试覆盖组合 PATCH 失败不应部分热生效；增加服务层测试覆盖只写触碰字段。

审查结论：

- 运行配置由 `SettingsService` 内存快照提供，普通请求热路径读取快照，不读 `config.json`。
- 配置修改和手动 reload 都通过写锁串行化，避免并发写入相互覆盖；成功写盘后才替换内存快照。
- `PATCH /api/settings` 只应用本次请求触碰的字段；本地手工修改 `config.json` 后需要显式调用 reload 才会进入内存快照。
- runtime 更新后会通过 `AppState::apply_runtime_settings` 替换目录/传输/IP 限流器、任务运行参数、搜索开关、回收站和审计策略；已开始的请求和任务继续按开始时持有的配置执行。
- startup 配置可以在线保存，但只影响下次启动；`activeStartup` 与 `restartPendingFields` 能展示当前进程和已保存配置的差异。
- 环境变量锁定字段会在补丁校验阶段返回 `409 CONFLICT`，不会被设置接口覆盖。

后续统一优化记录：

- `config.json` 写入是同目录临时文件加 rename；Windows 下目标存在时会先删除再 rename，存在极短窗口。主要部署环境是 Linux Docker，当前可接受；如果以后 Windows 作为主要部署目标，可评估使用更强的替换语义。

### 第七轮：认证与 API 错误

已处理：

- 修复公开认证接口缺少单 IP 并发保护的问题：`/api/auth/login`、`/api/auth/setup`、`/api/auth/session` 现在也会经过 IP 并发限制，避免登录校验和首次设置接口在局域网内被并发请求放大 CPU 压力。
- 修复首次设置管理员密码的并发初始化窗口：`AuthStore::initialize_admin_password` 在写锁内再次检查是否已初始化，两个并发首次设置请求最多只有一个成功，不会让后到请求覆盖先到密码。
- 收紧 `auth.json` 写入一致性：认证存储现在先写临时文件并 rename 成功，再替换内存密码快照；写盘失败不会造成“接口失败但当前进程密码已改变”。
- 统一 API 默认错误响应：无效 JSON/请求提取失败、登录后的未知 API 路径、错误 HTTP 方法会返回结构化 JSON 错误，前端可继续依赖 `code/reason/message/params`。
- 未知 `/api/*` 路径仍遵守认证边界：未登录先返回 `401 AUTH_REQUIRED`，登录后返回 `404 API_ROUTE_NOT_FOUND`。
- 增加 API 回归测试覆盖公开认证接口 IP 限流、未知 API 认证优先、方法不支持 JSON 错误、无效 JSON 结构化错误。

审查结论：

- 公开接口范围保持为 `GET /api/`、`GET /api/health`、`GET /api/ready`、`POST /api/auth/setup`、`POST /api/auth/login`、`GET /api/auth/session`；其余 API 默认要求登录。
- 会话仍为内存态单管理员会话，修改密码或首次设置后会清理旧会话；服务重启后需要重新登录，符合轻量局域网边界。
- 错误响应主结构保持 `{ code, reason, message, params? }`；`reason` 继续作为前端国际化主 key，`message` 保持中文兜底。
- 公共就绪检查仍不要求认证，便于 Docker 健康检查；详细运行指标 `/api/metrics` 仍要求认证。

后续统一优化记录：

- 当前会话 token 只存内存，服务重启会全部失效；这是轻量单管理员边界下的简单取舍。后续如果用户强烈需要“重启后保持登录”，再评估持久化会话，但需要同步考虑失效和重置机制。
- Cookie 当前使用 `HttpOnly`、`SameSite=Lax`、`Path=/`，未默认加 `Secure`；局域网 HTTP 和 Docker 本地测试更方便。若以后支持反代 HTTPS，可增加“信任 HTTPS/启用 Secure Cookie”的启动配置。
- 登录失败已增加内存级短暂冷却：同一来源连续失败达到阈值后返回 `429 LOGIN_FAILURE_COOLDOWN`，成功登录会清理该来源失败记录；不引入账号体系或持久化状态。

### 第八轮：搜索、收藏、审计、健康检查

已处理：

- 修复搜索索引热禁用与后台重建的状态一致性问题：禁用索引会取消当前重建、清空索引条目并保持 `disabled` 状态；后台扫描结束后不会再把状态覆盖回 `idle/failed`。
- 修复搜索取消状态的竞态窗口：取消接口在同一段状态更新中推进重建 token 和 `cancelled` 状态，避免重建收尾阶段交错写回。
- 搜索重建现在跳过符号链接和特殊文件，不再为符号链接目标读取元数据，保持与路径安全审查中的符号链接边界一致。
- 修复收藏夹写入一致性：创建、更新、删除、重排都会先构造新列表并写入 `favorites.json` 成功后再替换内存快照；写盘失败不会污染当前进程状态。
- 收藏夹 JSON 写入改为同目录临时文件加 rename，减少写入中断导致的文件损坏窗口。
- 增加单元测试覆盖搜索跳过符号链接、重建中禁用索引保持 `disabled`、收藏夹写盘失败不改变内存顺序。

审查结论：

- 搜索索引默认关闭，只有显式重建或启动配置开启重建时才递归扫盘；增量更新只处理单路径或内存前缀，不会为了普通搜索请求读盘。
- 搜索和最近文件接口只读内存索引，并通过 `maxDirPageSize` 限制单次返回规模；当前适合轻量局域网场景。
- 收藏夹默认列表不检查磁盘存在性，只有 `check=true` 才逐项解析路径，符合避免无意义磁盘操作的边界。
- 审计日志只记录登录、写入、上传、删除、恢复、任务和配置类操作；高频目录读取、预览、下载不写审计，已有 API 测试覆盖。
- 审计日志支持大小轮转和保留数量，启动时只读取当前日志大小，不扫描正文；清理旧轮转文件需要显式调用或在轮转时触发。
- 健康检查和就绪检查保持公开且轻量；详细指标 `/api/metrics` 仍要求认证。

后续统一优化记录：

- 搜索索引仍是内存索引，服务重启后需要手动或按配置重建；这符合当前“避免启动无意义扫盘”的边界。若以后需要重启后保留索引，再评估可选 SQLite 索引。
- 搜索重建会索引目录和文件的基础元数据；大树重建仍是显式操作，后续如需要更细控制，可以增加最大索引文件数或扫描路径范围，默认仍不自动扫全盘。
- 收藏夹数量预期较小，写入时 clone 整个列表和完整 JSON 覆写可接受；如果未来收藏数量异常增长，再考虑更细粒度的存储格式。
- 就绪检查会探测配置、认证、收藏、映射、回收站、审计和静态入口文件的基本可读写性；它不会检查所有挂载点是否在线，避免健康检查变成高频扫盘。

### 第九轮：启动、部署与静态资源就绪

已处理：

- 修复健康检查地址固定使用 `127.0.0.1` 的问题：`web-file-browser --healthcheck` 现在会从实际监听配置生成检查地址，`0.0.0.0` 和 `::` 会映射到本机 loopback，显式监听地址则保持原值。
- 修复 IPv6 监听地址解析问题：`AppConfig::socket_addr` 现在先解析 `bindAddress` 为 IP 再组合端口，支持 `::` 这类 IPv6 地址。
- 收紧 `/api/ready` 静态资源检查：不再只确认静态目录存在，还会确认 `index.html` 可读，避免容器健康但 Web 页面入口缺失。
- 测试辅助应用会生成最小 `static/index.html`，保持就绪检查测试和真实部署语义一致。

审查结论：

- Dockerfile 继续使用三阶段构建，运行镜像不依赖 `curl`，健康检查由应用二进制完成，符合轻量运行镜像边界。
- Compose 默认把 `/app/data`、`/mnt/files` 作为持久化和业务文件挂载，UID/GID 使用数字用户，避免构建期用户组冲突。
- Windows 打包脚本会生成独立测试目录、默认 `data/config.json` 和 `run-windows.bat`，并保护仓库根和磁盘根不被作为输出目录。
- 启动期配置路径仍只在启动时解析，普通请求不读配置文件；在线保存 startup 字段后需要重启生效，和前端设置页契约一致。

后续统一优化记录：

- 静态资源 fallback 已补充专项测试：普通前端深链接返回入口 HTML，登录后的未知 `/api/*` 仍返回结构化 JSON 404。
- Windows 打包脚本仍是测试包定位，不做服务安装、托盘或自启动；这些属于后续运维发布能力，不进入当前轻量后端边界。
